# 飞书知识库生产线 — Claude 项目说明

## 项目概述

基于飞书 + Claude Code 的专题知识库自动化生产系统。将外部资料经 AI 结构化处理后写入飞书多维表格，再聚合升级为专题知识库成品页面。

**GitHub 仓库**：`xiwenran/lark-knowledge`

---

## 核心架构

```
外部资料 → AI结构化处理 → 飞书多维表格 → 聚合升级 → 专题知识库成品
         (收件入表)                    (升级为页面)
```

**两段式工作流**：
1. **收件入表**：外部资料经 AI 提取关键信息后写入多维表格（结构化存储）
2. **升级成品**：从表格中筛选高价值记录，聚合生成知识库页面（内容产出）

**三层分类体系**：专题 → 资产形态 → 具体条目

## 开发约定

### 配置管理
- `config.json` 统一管理飞书 token、表格 ID、知识库空间 ID 等
- **禁止**在 skill 代码中硬编码飞书凭证或资源 ID

### 工作流规范
- 严格遵循两段式工作流：先收件入表，再升级为知识库页面
- 收件时必须完成 AI 结构化提取，不允许原文直接灌入
- 升级时按资产形态模板生成正文，自动调用排版美化

### 内容安全
- 排版不改文字（只增加颜色、callout、标题样式，绝不修改原始文字）
- 推送前检查是否包含敏感信息（token、个人路径等）

### 分类体系
- 新条目必须归入：专题 → 资产形态 → 具体条目 三层结构
- 新增专题需同步更新 config.json 和飞书规范文档

## 任务派发约定（主会话不自做清单）

本节为 Echo 三模型分工规则在本项目的具体落地。

### 禁止主会话自做

- **撰写知识库页面正文**：`/升级` skill 产出的页面内容，由 Sonnet 子代理生成，主会话不自写
- **撰写 AI 结构化提取内容**：`/收件` skill 的结构化字段提取，由 Sonnet 子代理完成
- **批量修改 skill 代码**（`.py` 文件，超过 10 行）：派 Codex 执行
- **撰写排版规则段落 / SKILL.md 更新**（超过一个段落）：派 Sonnet 子代理

### 允许主会话自做

- 调用 skill、协调流程顺序、给用户解释问题
- 决策哪些记录值得升级（判断类，非执行类）
- 审核子代理产出，提出修改意见
- ≤10 行的代码小改（参数调整、单行 bug 修复）
- 向用户展示摘要报告（不写入文件）

### Skill 内部模型约定优先

进入任何 skill 流程前，先扫 SKILL.md 的模型/工具约定段落（如"翻译用 Doubao"、"图生成走 Codex"）。
有约定 → 按 skill 约定执行，不被通用「文本→Sonnet」原则覆盖。
无约定 → 走全局三模型分工规则。

## ⚡ Echo 会话检查清单（每次任务完成前必查）

完成任何实质性改动后，**在报告完成之前**，按顺序检查：

**① 高风险改动？→ 冷眼审查**
- 触发条件：发布逻辑 / 写入文件或数据库 / 防重复机制
- 操作：说"我将触发冷眼审查"，用 Agent 派独立子代理只看 diff + 需求

**② 方向性问题出现？→ 主动建议圆桌讨论**
- 触发条件：项目定位、产品路线、架构主线、商业方向、长期维护策略；用户表达"要不要 / 像 A 还是像 B / 感觉但不确定"；或主会话自己准备列出 ≥2 个互斥方向
- 操作：在回答之前先停下，说"这是方向性问题，建议三省讨论。可以输入「圆桌讨论：[问题]」触发多 Agent 分析。"
- 注意：主动建议圆桌不等于擅自执行圆桌。用户确认后才执行完整 roundtable。方向性选项和清单式独立项判断不清时，默认建议圆桌。

**③ 有实质改动？→ Obsidian 捕获**
- 触发条件：新功能完成 / 重构 / bug 根因找到 / 推送 GitHub
- 操作：展示 changelog 草稿，等用户确认后写入 `~/Obsidian/PersonalWiki/项目/lark-knowledge/changelog/`

**④ git push 前 → 脱敏扫描**
- 检查：`/Users/用户名/`绝对路径、token、邮箱、AppID
- 发现敏感内容：立即停下告知用户，修改后再推
